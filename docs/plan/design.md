# Design: Not-To-Do

## Problem Statement

A habit-breaking app where users commit to things they want to stop doing. Every day they resist = +1 streak. Break the rule = back to Day 0. Future: challenge friends with a share link to quit together.
All streak counts in product copy use zero-based labels: `Day 0`, `Day 1`, `Day n`. Never `1 day / 2 days` or `1天 / 2天`.

## What Makes This Cool

- **Dual mechanic**: Check-in (consciousness) and resistance (discipline) are independent. Three states: no check-in (unconscious), resisted (had urge but didn't act), failed (had urge and gave in).
- **Hall of Shame**: Not just celebrating success. See who's on Day 0 in your challenge group. Public failure among friends is 10x stronger motivation than public success.
- **Small group accountability**: 5-6 person friend teams. Private or public challenges. Each challenge = one bad habit.

## Constraints

- **Progressive auth**: App works fully without login (anonymous session). Login is optional and saves/persists your records across devices. Both modes coexist permanently.
- Tech: Next.js 14 (App Router) + Drizzle ORM + pure PostgreSQL
- No Prisma, no Supabase
- Hosting: CapRover (self-hosted), plain PostgreSQL container
- Max 3 active not-to-do items per user (configurable later)

## Premises

1. One challenge = one bad habit. Users can join max 3 challenges (future configurable).
2. Three check-in states: no check-in (unconscious), resisted (had urge but didn't act), failed (had urge and gave in).
3. Hall of Shame = who is on Day 0 within a challenge.
4. Challenges can be public or private (invite link).
5. Streak resets to 0 only on "failed". No check-in = no penalty, no gain (frozen).

## Approaches Considered

### Approach A: Full Platform (DESIGN.md original)
All phases 1-4 at once. OAuth, PWA, push notifications, challenges, leaderboards.
Effort: XL. Risk: High. Ships too late.

### Approach B: Local-First Single User
No auth, no backend. All data in localStorage. Pure client-side React app.
Effort: S. Risk: Low. Ships immediately.
This is the chosen approach.

### Approach C: Local-First with Schema Preview
Same as B for runtime, but V2 database schema documented and ready for migration.
Effort: S+. Risk: Low. Clear V2 migration path documented upfront.

## Recommended Approach

**Approach C: Local-First with Schema Preview.** V1 is pure client-side (localStorage, no backend). V2 adds Drizzle + PostgreSQL + auth + sync. The V2 schema is documented in this design doc so the migration path is clear, but V1 doesn't need any server-side setup at all. Zero infrastructure cost for V1.

## V1 Scope (Progressive Auth: works without login, login persists data)

### Key Architecture Decision: Local-First, Sync on Login
- **Without login**: All data stored in browser localStorage. No server-side storage. Fully functional offline.
- **With login (V2)**: Data syncs to PostgreSQL. Persistent across devices.
- **Login = upgrade**: The app is fully functional from first visit, zero friction. Login is for persistence and multi-device.
- V1 limitation: clearing browser data loses records. Acknowledged. This is the tradeoff for zero-friction onboarding.

### Data Layer (V1: localStorage)
```
// localStorage keys
ntd_items: JSON array of NotToDo items
ntd_checkins: JSON array of check-in records

// Each item
{ id, title, description, streak, bestStreak, lastCheckin, isActive, createdAt }

// Each check-in
{ id, notToDoId, date, status: 'resisted' | 'failed', createdAt }
```

### Data Layer (V2: PostgreSQL via Drizzle)
```sql
users
  id          UUID PK
  email       VARCHAR UNIQUE
  name        VARCHAR

  not_to_do_id  UUID FK -> not_to_dos.id
  date          DATE

challenges
  id             UUID PK
  title          VARCHAR
  description    TEXT
  slug           VARCHAR UNIQUE
  creator_id     UUID FK -> users.id

  id              UUID PK
  challenge_id    UUID FK -> challenges.id
  user_id         UUID FK -> users.id
  not_to_do_id    UUID FK -> not_to_dos.id
```

### Pages
- `/` Dashboard: today's check-in list
- `/items` All not-to-do items
- `/items/new` Create item
- `/items/[id]` Item detail + check-in history

### API Routes (V1: client-side only, no server)
All CRUD operations read/write to localStorage directly.
No API routes needed in V1. Next.js renders client components.

### API Routes (V2: server-side with Drizzle)
- `GET/POST /api/items` List / create
- `PATCH/DELETE /api/items/[id]` Update / archive
- `POST /api/checkin` Submit daily check-in (resisted | failed)
- `GET /api/checkin/[id]` Check-in history
- `POST /api/auth/register`, `POST /api/auth/login`
- `POST /api/sync` Upload localStorage data to server on first login

### Check-in Logic
- Dashboard shows today's active items
- Each item shows a clear primary action: "Day +1"
- Each item also shows a secondary action: "破戒，重置 Day"
- Resisted: streak + 1, update best_streak if beaten
- Failed: streak = 0
- No check-in: no change (frozen)
- One check-in per item per calendar day
- Once a user checks in for today, today's state is locked and cannot be changed again until tomorrow
- Day resets at midnight in user's local timezone (client-side calculation)

## V2 Scope (Auth + Multi-user + Challenges + Hall of Shame)

### Auth Strategy: Claim-and-Upgrade
- Anonymous user visits, gets session cookie, data stored with session_id
- User clicks "Save my progress" -> login/register flow
- On first login: all data with their session_id gets user_id assigned
- App works identically with or without login from Day 0

### New Schema
```sql
users (id, email, name, avatar_url, created_at)
challenges (id, not_to_do_template_title, description, slug, creator_id, is_public, created_at)
challenge_participants (id, challenge_id, user_id, not_to_do_id, joined_at)
```

### New Features
- Auth (NextAuth v5, credentials + Google OAuth)
- "Save progress" CTA for anonymous users
- Create challenge from any not-to-do item
- Share link to join (public or private)
- Hall of Shame: per-challenge view of who's on Day 0
- Ranking: resistance ratio, streak, consciousness score
- PWA + Web Push notifications
- Confession on failure (optional text)
- Max 3 active challenges per user

## What I noticed about how you think

- You corrected "抵抗" to a very precise definition: "想做了，但我沒有做". That's specific. Most people would say "didn't do it." You're distinguishing between "didn't think about it" and "thought about it and chose not to." That's the insight that makes this app different from every streak tracker.
- You wanted Hall of Shame, not Hall of Fame. That's a product instinct most founders miss. Celebrating success is safe. Publicizing failure is memorable.
- You constrained scope early: "先做沒有登入的個人版本." Ship fast, add later. That's the right instinct.
